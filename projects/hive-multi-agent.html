Multi-Agent 기반 고객 지원 자동화 시스템

장애가 터지면 고객 문의는 하루 6,000건까지 치솟습니다. 이 급증을 사람 대신 흡수하기 위해 설계한 Graph 기반 Multi-Agent 오케스트레이션 시스템 — 문의 유형별로 Agent의 책임을 나누고, Supervisor가 적합한 Agent로 라우팅합니다. Interaction → Report → Action 3단계로 동작하며, 피크 시 Agent 기여율 34.4%를 기록하면서 2개 실서비스를 운영 중입니다.

02 Problem

실서비스 운영팀은 월 평균 1,000건, 장애 시에는 하루 6,000건까지 급증하는 고객 문의를 소수 인력으로 수작업 처리하고 있었습니다. 계정 분실, 결제 오류, 아이템 미지급 등 유형별 대응 절차가 달라 일관된 처리가 어려웠고, 피크 시에는 대응 지연이 불가피했습니다.

"문의 유형에 따라 대응 절차가 달라지는데, 이걸 사람이 매번 분기해서 처리하고 있다. 자동화하려면 분기 자체가 지능적이어야 한다."

단일 LLM 프롬프트로 모든 유형을 처리하는 접근은 운영팀 인터뷰 단계에서 일찍 기각되었습니다. 유형별 데이터 소스·승인 절차·Action API가 서로 달랐고, 하나의 거대한 프롬프트로 묶을수록 품질이 나빠지는 구조였기 때문입니다.

03 Approach

Hive — Graph 기반 Multi-Agent 오케스트레이션

유형별 대응 절차를 독립된 Agent로 분리하고, 상위의 Supervisor Agent가 문의를 라우팅하는 구조로 설계했습니다. Agent 간 관계를 Graph로 모델링하여 Agent 추가·변경 시 전체 시스템을 다시 설계하지 않아도 되도록 했습니다.

  • Supervisor Agent — 사용자 문의를 분석하여 적합한 하위 Agent(계정분실, 결제, 아이템 미지급 등)로 hand-off.
  • Worker Agents — 각 Agent는 Interaction → Report → Action의 3단계로 동작. 사용자와의 대화로 정보를 수집하고(Interaction), 처리 가능 여부와 근거를 구조화한 뒤(Report), 실제 조치를 수행(Action)하는 명확한 책임 경계.
  • Admin 운영툴 — 비개발자(운영팀)가 직접 Agent 정의·프롬프트·정책을 생성·관리할 수 있는 인터페이스. 유형 추가가 개발 병목이 되지 않도록 하는 것이 핵심이었음.
  • Langfuse 트레이싱 — Supervisor → Worker hand-off, 각 단계의 LLM 호출, 최종 Action까지의 흐름을 전 구간 추적하여 품질 모니터링과 실패 원인 분석에 사용.

왜 Graph 구조인가

단순 트리가 아닌 Graph로 설계한 이유는, 실제 운영에서 Agent 간 공유 컨텍스트와 cross-referral이 빈번하게 발생하기 때문입니다. 결제 오류가 계정 이슈와 얽혀 있거나, 아이템 미지급이 장애 이벤트와 연결되는 등 단방향 분기만으로는 표현되지 않는 케이스가 많았습니다. Graph 구조는 새로운 Agent를 기존 노드 사이에 삽입하는 형태의 확장을 자연스럽게 허용합니다.

05 Impact · 2025.07 운영 리포트 기준

  • 자동처리율 ~56% — 문의 절반 이상을 운영팀 개입 없이 Agent가 종결.
  • 피크 시 Agent 기여율 34.4% — 평상시 약 14%에서, 장애 등 문의 급증 시점에 자동처리 비중이 크게 상승. 운영팀이 감당하기 어려운 피크를 Agent가 실제로 흡수.
  • 피크일 기준 2,319건 처리 — 단일 일자에 Agent가 종결한 문의 수.
  • 담당자 소통 단계 감소 — Report 단계에서 근거가 구조화되어 전달되므로, 에스컬레이션 시에도 운영자가 확인해야 할 맥락이 줄어듦.
  • 2개 실서비스 동시 운영 — 브라운더스트2(FAQ + CS 자동화), 피망 뉴맞고(FAQ).

06 Takeaways

  • "하나의 똑똑한 프롬프트"보다 "책임이 분리된 작은 Agent들" — 유형마다 데이터·승인·Action이 다른 도메인에서는 단일 LLM 프롬프트를 키우는 방향이 빠르게 한계에 부딪혔음. 책임 경계를 Agent 단위로 분리하고 Supervisor가 라우팅하는 구조가 품질·확장성 모두에서 우세.
  • Agent 생성권을 비개발자에게 넘긴 것이 지속성의 열쇠 — Admin 운영툴로 운영팀이 직접 Agent를 정의·수정할 수 있게 되자, 유형 추가가 개발 큐에 쌓이지 않고 운영 사이클 안에서 해결됨. 실제 자동처리율은 런칭 이후에도 계속 개선.
  • Langfuse 트레이싱은 "모니터링 도구"가 아니라 "설계 피드백 루프" — Supervisor hand-off 오류, Worker의 Report 누락 등 문제의 원인이 구간 단위로 드러나자, 설계 의사결정을 데이터 기반으로 내릴 수 있게 됨.
  • 피크 때 진가가 드러나는 시스템 — 평상시 14% / 피크 34.4%라는 숫자는, 자동화가 "평상시를 약간 편하게" 하는 것이 아니라 "피크의 대응 한계를 늘리는" 역할로 기능했음을 보여줌.

07 Transferable Value

본 프레임워크는 고객 문의가 유형별로 서로 다른 처리 절차를 가지는 도메인이라면 그대로 이식 가능합니다. 이커머스 CS(배송·환불·교환), 금융 고객지원(계정·거래·분쟁), SaaS 기술지원(결제·권한·버그 리포트) 등에서 Supervisor-Worker 분리와 NetworkGraph 확장성의 이점이 동일하게 적용됩니다.
