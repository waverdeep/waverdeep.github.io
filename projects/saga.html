Project · NeoChat PRO 내 "문서 지식베이스"

사업/운영 특화 생성형 지식 검색 시스템 (SAGA)

프로젝트별로 축적된 기획서·운영 리포트·마케팅 자료·계약서 등을 맥락에 맞게 검색·활용할 수 있도록 설계한 하이브리드 RAG 시스템. OpenSearch 기반 BM25 + Neural Vector 검색, Docling / Vertex AI 문서 변환, MCP 프로토콜, RAGAS 품질 검증, 도메인 Custom Dictionary까지 포함한 End-to-End 파이프라인으로 전사 Live 운영 중.

2025.06 — 2025.12 시스템 설계 · 검색 파이프라인 · 전처리 전사 Live 요구사항 커버리지 ~90%

01 Metadata

2025.06 — 2025.12 (6~7개월) · 개발 완료 → 전사 Live

02 Problem

Mimir를 통한 전사 통합 검색은 이미 제공되고 있었지만, 프로젝트별로 축적된 문서(기획서, 운영 리포트, 마케팅 자료, 계약서 등)를 맥락에 맞게 검색·활용하는 기능은 부족했습니다. "이 프로젝트 안에서만, 최신 버전 기준으로" 찾고 싶다는 요구가 많았지만, 전사 검색은 입자가 너무 거칠었습니다.

"전사 검색은 넓이를, SAGA는 깊이를 담당한다."

게다가 기획서·계약서 같은 문서는 PDF·PPT·표·이미지가 섞여 있어 단순 텍스트 추출로는 품질이 나오지 않았습니다. 도메인 용어(사내 프로젝트 코드명, 약어)는 일반 tokenizer로 잘리면 검색 결과가 크게 저하되었고, RAG 품질을 지속적으로 측정·개선할 수 있는 루프도 함께 필요했습니다.

03 Approach

Hybrid Search — BM25 + Neural Vector

OpenSearch 기반으로 BM25(어휘 일치)와 Neural Vector(의미 유사도)를 50:50으로 결합했습니다. 사내 문서는 고유명사·약어 비중이 높아 BM25 단독이 강하지만, 의도 기반 질의(예: "작년 대규모 업데이트 때 무슨 이슈가 있었지?")에는 Neural이 필요합니다. 둘 다 쓰는 게 아니라 어느 비율로 섞을지가 핵심이었고, 실제 질의 로그로 튜닝했습니다.

Document Intelligence — Docling Serve (primary) / Vertex AI (fallback)

PDF·PPT·표·이미지가 섞인 문서를 구조 보존 상태로 변환하기 위해 Docling Serve를 primary로 사용하고, 복잡한 레이아웃이나 이미지 비중이 큰 문서에는 Vertex AI Gemini를 fallback으로 두었습니다. 변환 결과는 markdown으로 통일하여 downstream의 chunking·임베딩 파이프라인이 형식에 무관하게 동작하도록 했습니다.

Preprocessing Pipeline — saga-research-cli

문서에서 metadata · markdown · summary · keywords를 자동 추출 → OpenSearch ML 파이프라인으로 임베딩·인덱싱하는 CLI 도구를 개발했습니다. summary·keywords는 검색 품질뿐 아니라 사용자에게 보여지는 미리보기·필터링 UX에도 활용됩니다.

OpenSearch Custom Dictionary

사내 프로젝트 코드명, 약어, 도메인 용어를 반영한 tokenizer / filter / analyzer를 커스터마이즈했습니다. 일반 한국어 tokenizer는 고유명사를 깨뜨리기 쉬워 BM25 쪽 품질이 크게 저하되었는데, Custom Dictionary 적용 후 실제 질의에서 명사구 리콜이 눈에 띄게 개선됐습니다.

MCP 프로토콜 — AI Agent가 직접 질의

NeoChat PRO와 Claude Desktop 같은 MCP 클라이언트가 SAGA를 도구로 직접 호출할 수 있도록 search_project_docs, get_document_content 등의 MCP tool을 노출했습니다. 이로써 Agent는 대화 중에 "프로젝트 X의 기획서를 찾아서 근거로 쓰는" 동작을 자연스럽게 수행할 수 있습니다.

RAGAS 기반 품질 검증 루프

문서별 Q&A pair를 자동 생성하고 RAGAS로 검색·생성 품질 점수를 측정하는 회귀 테스트 체계를 구축했습니다. 문서가 추가될 때마다 품질 지표를 재측정하여, 임베딩 모델·검색 파라미터·전처리 옵션을 바꾸는 의사결정을 감이 아니라 숫자로 할 수 있게 만든 것이 핵심.

04 4단계 구현 계획

  • Phase 1 — MVP: 파일 업로드 & RAG
  • Phase 2 — Mimir 연동: JIRA / Wiki 통합, 프로젝트별 필터링
  • Phase 3 — Vertex AI 연동: Google Workspace 데이터 통합
  • Phase 4 — 외부 시스템 연동: 커뮤니티분석툴 · 태블로 · 법률자문시스템

05 Impact

  • NeoChat PRO 내 "문서 지식베이스"로 전사 Live 운영 — 임직원이 일상적으로 사용하는 지식 검색 접점.
  • Phase 1 요구사항 커버리지 약 90% — MVP 범위를 실 사용자 관점에서 검증.
  • MCP 프로토콜로 Agent 직접 접근 — NeoChat PRO, Claude Desktop 등 MCP 호환 Agent가 SAGA를 도구로 활용 가능. RAG가 "별도 검색 UI"가 아니라 "Agent의 능력"이 됨.
  • 전사 AI 활용 전략 로드맵 명시 — 조직의 공식 AI 로드맵에서 지식 계층을 담당하는 컴포넌트로 자리매김.

06 Takeaways

  • "Neural이 BM25를 이긴다"는 통념은 도메인에 따라 다르다 — 고유명사·약어 비중이 높은 사내 문서에서는 BM25가 여전히 강력했고, 비율 튜닝과 Custom Dictionary가 모델 교체보다 더 큰 품질 이득을 줌.
  • 문서 변환 품질이 곧 RAG 품질 — 아무리 좋은 임베딩·검색을 써도 Docling 단계에서 표가 깨지면 downstream이 복구할 수 없음. Document Intelligence 계층에 시간 투자한 것이 가장 큰 ROI를 냈음.
  • RAGAS 회귀 테스트가 의사결정 방식을 바꿈 — "이게 더 좋아진 것 같아요"라는 체감 기반 개선에서, "지표가 X에서 Y로 올랐다"는 숫자 기반 개선으로 팀의 대화 방식이 이동함.
  • MCP로 RAG를 "기능"이 아닌 "도구"로 노출한 것의 파급력 — 검색 UI를 만드는 대신 MCP tool을 노출하자, 별도 UX 없이도 NeoChat PRO의 모든 워크플로우가 SAGA를 쓸 수 있게 됨.
